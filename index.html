🛑 Section 1: ILP 的基本概念與效能公式 (Slides 1-2)

1. 什麼是管線化與 ILP？

  • 管線化的歷史意義：自 1985 年左右開始，世界上所有的處理器都開始採用管線技術（Pipelining）來大幅度提升處理器的效能。
  • ILP 的物理定義：指令之間所存在的「潛在重疊執行能力」，在計算機結構中就被稱為 Instruction-Level Parallelism (ILP)。

2. 開發 ILP 的兩大流派 (Approaches to Exploit ILP)

為了要把管線的效能壓榨到極限，主要有以下兩種截然不同的手段：

  • 動態方法（Dynamic approach）：完全依賴硬體（HW）在執行期（Runtime）即時偵測平行度並動態調整指令順序。
  • 靜態方法（Static approach）：完全依賴軟體/編譯器（SW）在編譯時期就把指令重排、拉平。

3. 管線效能的終極指標：Pipeline CPI 公式

考試若出選擇題或計算題，這個公式是整章的基石：

  • Ideal pipeline CPI：最完美的理想管線 CPI（通常為 1，在多重發射處理器中可以小於 1）。
  • Structural stalls：因硬體資源衝突產生的暫停。
  • Data hazard stalls：因資料還沒算好、前後指令資料相關產生的暫停。
  • Control stalls：遇到分支指令（Branch）猜不透、算不出來而產生的暫停。

🎯 Section 3: ILP 的開發範疇：基本區塊與迴圈 (Slide 4)

1. 基本區塊 (Basic Block) 的局限性

  • 基本區塊的定義：一段直線執行的程式碼，除了入口外沒有分支跳進來，除了出口外沒有分支跳出去。
  • 殘酷的現實：在一個 Basic Block 內部，可開發的 ILP 其實非常小。因為裡面的指令通常前後行高度相關。
  • 正確的策略：處理器必須跨越多個 Basic Blocks（Across basic blocks）去尋找獨立的指令來重疊執行。

2. 迴圈層級平行化 (Loop-Level Parallelism)

跨區塊開發最經典的例子就是迴圈。

// 投影片經典 C 語言範例
for (i = 1; i <= 1000; i = i + 1)
    x[i] = x[i] + y[i];
  • 知識點解說：在單一次迴圈內部（例如算 x[1]），指令是有前後依賴的。但是，計算 x[1] 的這整套動作，跟計算 x[2] 的那整套動作，彼此之間是完全獨立的！這就叫 Loop-level parallelism。
  • 硬體壓榨絕招：我們後續可以使用 Vector instructions（向量指令） 來完美開發這種迴圈層級的平行度。

投影片明確指出，程式中存在三種依賴性：

  1. 資料依賴 (Data dependences)
  1. 名稱依賴 (Name dependences)
  1. 控制依賴 (Control dependences)

1. 資料依賴 (Data Dependences) —— 又稱「真依賴 (True Dependence)」

  • 定義：如果指令 j 要用到指令 i 產生的結果（算出來的值），那麼指令 j 就資料依賴於指令 i。
  • 傳遞性：若指令 j 依賴於 k，而 k 又依賴於 i，則 j 傳遞依賴於 i。
  • 對管線的衝擊：兩道有資料依賴的指令，絕對不能同時執行，也絕對不能完全重疊管線。它們決定了資料計算的先後順序，並且設定了平行度的上限。
  • 如何克服：要麼在硬體上設法避免它引發 Hazard（例如 Forwarding），要麼透過軟體代碼轉換（Transforming the code）來消除它。

🔎 投影片經典 MIPS 程式碼拆解 (Slide 6)

程式碼片段

Loop: L.D      F0, 0(R1)      ; 指令 1: 從記憶體載入資料到 F0 暫存器
      ADD.D    F4, F0, F2     ; 指令 2: 計算 F4 = F0 + F2
      S.D      F4, 0(R1)      ; 指令 3: 把 F4 的結果存回記憶體
      DADDUI   R1, R1, #-8    ; 指令 4: 將指標 R1 減 8
      BNE      R1, R2, Loop   ; 指令 5: 若 R1 跑完則跳出，否則繼續迴圈
  • Data Dependence 深度解析：
    1. ADD.D 資料依賴於 L.D，因為它需要讀取 L.D 寫入的 F0。
    1. S.D 資料依賴於 ADD.D，因為它需要把 ADD.D 算好的 F4 存入記憶體。
    1. BNE 資料依賴於 DADDUI，因為它需要讀取 DADDUI 更新後的 R1 來做判斷。
  • 資料依賴的流向：資料依賴既可以透過暫存器（Registers）流動，也可以透過記憶體（Memory）流動。

2. 名稱依賴 (Name Dependences) —— 又稱「假依賴」

  • 定義：兩道指令同時使用了同一個暫存器或記憶體位址（這個名字 Name），但是這兩道指令之間實際上沒有任何資料的流動或交換。
  • 核心本質：這純粹是因為暫存器不夠用，程式設計師或編譯器「剛好重複用了同一個名字」而造成的卡頓。只要把名字換掉（Renaming），平行度就出來了。

名稱依賴在程式順序（Program order）中分為以下兩種：

🅰️ 反依賴 (Antidependence)

  • 定義：後面的指令 j 要去寫入（Write）某個暫存器，但是排在前面的指令 i 還要讀取（Read）這個暫存器的舊值。順序絕不能顛倒，否則前面指令會讀到錯誤的新資料！
  • 投影片經典範例 (Slide 9)：程式碼片段
    add  r1, r2, r3   ; 指令 i: 讀取 r2, r3 的值，算好寫入 r1
sub  r2, r4, r5   ; 指令 j: 算好結果，寫入覆蓋 r2
  • 深度解析：sub 指令反依賴（Antidependent）於 add 指令。因為 add 還要讀取 r2 的舊值，sub 絕對不能搶先跑完去更新 r2。

🅱️ 輸出依賴 (Output Dependence)

  • 定義：排在前面的指令 i 和排在後面的指令 j 都要寫入（Write）同一個暫存器位置。執行順序不能顛倒，否則最後留下來的暫存器數值會錯誤。
  • 投影片經典範例 (Slide 10)：程式碼片段
    add  r1, r2, r3   ; 指令 i: 寫入 r1
sub  r1, r4, r5   ; 指令 j: 也寫入 r1
  • 深度解析：sub 指令輸出依賴（Output dependent）於 add 指令。它們都在搶著改寫 r1，必須確保最後留在 r1 裡的是程式順序上較晚的 sub 的結果。
💥 Section 5: 資料危險 (Data Hazards) (Slide 11)

投影片指出：「只要指令之間存在依賴性（Dependences），管線在執行時就有可能發生 Hazard（危險）。」

考試時，Dependence 是指程式碼本質上的相關性（靜態概念），而 Hazard 是指管線執行時會引發的硬體衝突與停頓（動態概念）。

Data Hazards 共有以下三種：

  1. RAW (Read After Write)：後面要讀，前面還沒寫。這對應到資料依賴（Data dependency）。
  1. WAW (Write After Write)：後面要寫，前面也想寫，順序不能亂。這對應到輸出依賴（Output dependency）。
  1. WAR (Write After Read)：後面要寫，前面還要讀，前面沒讀完後面不能先寫。這對應到反依賴（Anti dependency）。
🚦 Section 6: 控制依賴與投機執行 (Slides 12-14)

1. 控制依賴的定義與鐵律

控制依賴是由分支指令（如 if-else）所引起的。

C

// 投影片 C 語言程式碼 (Slide 12)
if p1 {
    S1;   // S1 控制依賴於 p1
}
if p2 {
    S2;   // S2 控制依賴於 p2
}

為了確保程式邏輯正確，硬體在維持控制依賴時必須遵循兩大鐵律：

  • Program order（程式順序）：在分支還沒決定前，不能把裡面的指令搬到分支外面提早執行。
  • Branch detection（分支偵測）：必須精確處理分支結果。

程式碼片段

📑 計算機結構：Chapter 3 中段核心——編譯器技術與迴圈展開 (Slides 15 - 35)
🛑 1. 編譯器暴露 ILP 的基本原理 (Slide 15)

為了維持管線的滿載執行，編譯器必須設法找出彼此獨立、可以並行重疊的指令 。為了完美避免管線停頓（Pipeline Stall），一個存在依賴性的指令，必須與其「來源指令」在時鐘週期上拉開足夠的距離，這個距離必須大於或等於來源指令的管線延遲（Pipeline Latency） 。

編譯器進行這種指令排程（Instruction Scheduling）的能力，高度取決於：

  • 程式碼本身可開發的指令層級平行度（ILP）。
  • 硬體管線架構所設定的運作延遲（Latencies） 。
📊 2. 浮點數操作延遲表 (Slide 16)

在進行任何排程計算題前，必須嚴格遵守題目給定的硬體環境假設 ：

  • 分支延遲（Branch delay）：1 cycle 。
  • 浮點數 Load 到 Store（FP load to store）：0 cycle（因硬體 Forwarding 技術支援，無延遲） 。
  • 整數 Load 延遲（Int load delay）：1 cycle 。
  • 整數 ALU 操作（Int ALU op）：0 cycle（無延遲） 。
  • 結構危險（Structural hazards）：假設完全不存在 。

🅱️ 經編譯器優化排程後的管線狀態 (Slide 19)

優化策略是透過調整指令順序，利用不相關的指令填滿延遲空隙 ：

Loop: L.D      F0, 0(R1)      ; Cycle 1
      DADDUI   R1, R1, #-8    ; Cycle 2 -> 提早計算 R1，填補 L.D 的延遲槽 
      ADD.D    F4, F0, F2     ; Cycle 3 -> 完美避開 L.D 延遲限制 [cite: 139]
      stall                   ; Cycle 4 -> ADD.D 到 S.D 需要 2 週期，此處仍餘 1 個 stall [cite: 140]
      BNE      R1, R2, Loop   ; Cycle 5 -> 提早發射分支指令 [cite: 141]
      S.D      F4, 8(R1)      ; Cycle 6 -> 關鍵：將 Store 移至分支延遲槽（Branch-delay slot）！

📊 效率統計：單次迭代縮短至 6 個時鐘週期 。

⚠️ Notion 必記考點：因為 DADDUI 在第 2 週期提早將 R1 減了 8 ，最後一行 S.D 的偏移量（Offset）必須相應地從 0(R1) 修改為 8(R1) ，否則會寫入錯誤的記憶體位址 。

🔁 4. 迴圈展開技術 (Loop Unrolling) (Slides 20 - 21)
  • 核心定義：迴圈展開是一種用來增加指令層級平行度（ILP）的簡單且有效手段 。其作法是將迴圈主體複製（Replicate）多次，並同步調整迴圈的終止判斷程式碼 。
  • 主要優勢：能夠完全消去多餘的控制分支，並允許來自不同迭代（Iterations）的指令被混合編排在一起，極大化管線重疊效率 。

數學模型
若迴圈上限為 n，我們計畫將其展開成 k 個主體副本 ：

  1. 第一段小迴圈：執行 (n mod k) 次，內部使用原始未展開的迴圈主體 。
  1. 第二段主迴圈：執行 (n/k) 次，內部包裹著展開後的 k 份巨型迴圈主體 。
  1. 對於大型的 n 值，處理器絕大部分的執行時間都會集中在第二段已展開的精簡主體中 。

🔍 6. 編譯器變更指令順序的合法性科學 (Slide 26)

編譯器若要成功進行迴圈優化，必須具備以下分析能力並達成特定假設 ：

  1. 位址安全調動：能精確判定將 S.D 移至 DADDUI 與 BNE 後方執行是合法的，並能準確計算出 Offset 的校正量 。
  1. 分析獲利性：能正確評估進行迴圈展開確實能帶來效能獲利 。
  1. 消除假依賴：必須靈活指派不同的暫存器，以避開無謂的名稱依賴約束 。
  1. 移交控制開銷：能完美剔除多餘的測試與分支指令，並精準重構邊界終止代碼 。
  1. 記憶體去別名分析（Alias Analysis）：能明確判斷不同的 Load 與 Store 指令之間所存取的記憶體位址是互相獨立、可以任意互換順序的 。
  1. 保持語意一致：在重排整個代碼序列時，仍能完美保留程式原本的真資料依賴關係，確保運算結果百分之百正確 。
💣 7. 名稱依賴如何扼殺平行度？（對照組實驗） (Slides 29 - 31)

實驗：如果展開 4 次，但「故意不換暫存器名字」會怎樣？ (Slide 29, 30)
代碼如下（大家都重度依賴 F0 與 F4） ：

Loop: L.D      F0, 0(R1)
      ADD.D    F4, F0, F2     
      S.D      F4, 0(R1)      
      ; ─── 跨迭代間爆發嚴重的名稱依賴 ───
      L.D      F0, -8(R1)     ; ❌ 產生 WAR 阻礙！因為它想改寫 F0，但上方的 ADD.D 還要讀取 F0 的舊值！
      ADD.D    F4, F0, F2     ; ❌ 產生 WAW 阻礙！因為它與第一組的 ADD.D 同時在爭奪寫入 F4 的順序！
      S.D      F4, -8(R1)
  • 悲慘結果：因為跨迭代之間卡了死結一般的 WAR（反依賴）與 WAW（輸出依賴） ，這些指令完全無法被交錯打亂重排，管線只能像未展開前一樣原地乾等，平行的優勢被徹底抹殺 。
  • 唯一解法：透過 Register Renaming（暫存器重新命名） 換成不同的暫存器名字（如 F6, F8 … ），即可將這些假依賴當場消滅，解放平行度 。
🛑 8. 迴圈展開的三大物理極限 (Slides 32 - 35)

迴圈展開不可能無限無限進行，它受到三個現實條件的強力制約 ：

  1. 邊際效益遞減 (Overhead Amortization Limit)
    • 當我們將迴圈展開 4 次時，在 14 個執行週期中，只有 2 個週期是控制開銷（DADDUI 與 BNE），平均每次原始迭代分攤到 0.5 cycle 的開銷 。
    • 如果進一步激進地展開到 8 次，分攤開銷僅能從 0.5 cycle 微幅降至 0.25 cycle 。這代表大費周章展開代碼換來的效能回報會快速遞減 。
  1. 程式碼體積限制 (Code Size Limitations)
    • 代碼體積若過度膨脹，在空間極度珍貴的嵌入式系統中是無法被接受的 。
    • 在高性能處理器中，過大的代碼會直接塞爆指令快取，導致指令快取缺失率（I-Cache Miss Rate）大幅上升 ，CPU 頻繁為了去主記憶體抓指令而陷入嚴重卡頓，得不償失 。
  1. 編編器限制與暫存器壓力 (Register Pressure)
    • 暫存器壓力（Register pressure）是迴圈展開最致命的瓶頸 。為了壓榨 ILP 而將多組迭代的指令交錯編排，會導致「同時存活、等待被使用的變數值（Live values）」數量直線飆升 。
    • 一旦這些活變數的數量超出了硬體所能提供的實體暫存器總數，編譯器將無法為它們分配暫存器空間 。
    • 此時，編譯器被迫必須把部分變數資料溢出（Spill）寫回到速度極慢的記憶體中，這個動作產生的延遲會將迴圈展開帶來的效能優勢完全扣光 。
    • 備註：在不進行展開的狀況下，激進的指令重排會受到分支指令（Branches）的天然阻隔，活變數不容易累積，因此暫存器壓力極少成為問題 。

🛑 1. 靜態分支預測 (Static Branch Prediction) (Slides 36 - 41)

📄 核心概念與編譯器角色 (Slide 36)

  • 使用場景：靜態分支預測器有時會用於預期分支行為在編譯時期高度可預測的處理器中 。
  • 輔助功能：靜態預測也可以用來協助動態預測器 。
  • 優化管線危險：延遲分支（Delayed branches）會暴露出管線危險，編譯器可以藉此減少與該危險相關的懲罰（Penalty）。此外，在編譯時期若能準確預測分支，對於排程資料危險（Data hazards）也十分有幫助 。

📄 靜態預測方法一：永遠預測跳轉 (Predict-Taken) (Slide 38)

  • 原理：最簡單的靜態預測方案是直接將分支一律預測為跳轉（Taken）。
  • 平均效能：此方案的平均錯誤預測率（Misprediction rate）剛好等於不跳轉（Untaken）的分支頻率，在 SPEC 基準測試程式中該頻率為 34% 。
  • 缺點與波動：其正確率分布極度不均勻，錯誤預測率的範圍從不準確的 59% 到高度準確的 9% 都有可能 。

📄 靜態預測方法二：基於剖析資訊的預測 (Profile-Based) (Slides 39 - 41)

  • 原理：這是一種更準確的技術，它根據從先前運行（Earlier runs）中收集到的剖析資訊（Profile information）來預測分支 。
  • 科學根據：分支的行為在統計上通常呈現雙峰分布（Bimodally distributed），意即同一個分支指令的走向往往具有高度的一致性 。
  • 實測數據 (Slide 40，如下圖)：投影片展示了 SPEC92 基準測試在 Profile-Based 預測下的錯誤率 ：
    • doduc：5%
    • ear：6%
    • hydro2d：9%
    • mdljdp：10%
    • gcc：11%
    • compress / li：12%
    • su2cor：15%
    • espresso：18%
    • eqntott：22%
  • 效能對比 (Slide 41)：以「兩次錯誤預測之間成功執行的指令數（Instructions between mispredictions）」為指標（數值越高代表越精準）：
    • 在 doduc 中，盲猜 Predict-Taken 每 19 個指令就會錯一次，但 Profile-Based 可以撐到 250 個指令才錯一次。
    • 在 su2cor 中，Predict-Taken 撐 96 個指令，Profile-Based 則能達到 253 個指令。
    • 數據全面證明 Profile-Based 顯著優於無腦盲猜。
🛑 2. 動態硬體分支預測 (Dynamic Hardware Prediction) (Slides 42 - 50)狀態機四種狀態 (Slide 46)：

📄 硬體優化動機與阿姆達爾定律 (Slide 42)

  • 效能瓶頸：隨著我們試圖開發越來越多的指令層級平行度（ILP），控制依賴（Control dependences）會迅速成為限制效能的主因 。
  • 多重發射的需求：在一個每個週期能發射多道指令的處理器（n-issue processor）中，分支通常必須要被成功預測 。
  • 阿姆達爾定律的警告：當這類機器的潛在 CPI 越低時，控制阻礙（Control stalls）所帶來的相對衝擊（Relative impact）將會被放大 。

📄 動態預測的基本目標 (Slide 43)

  • 核心定義：動態預測方案的預測方向，完全取決於分支在執行期（Runtime）的即時行為 。
  • 主要目的：所有動態機制的終極目標，都是為了讓處理器能夠儘早判定（Resolve early）分支的最終結果 。
  • 帶來的效益：它能有效防止控制依賴在管線內部引發停頓（Stalls）。

📄 分支預測緩衝區 (Branch-Prediction Buffer, BPB) (Slide 44)

  • 硬體結構：BPB 是一塊微小的記憶體，直接使用分支指令位址的後半段低位元（Lower portion of the address）來進行索引（Index）。
  • 包含內容：該記憶體內部包含一個位元（Bit），用來記錄該分支最近是 Taken 還是 Not Taken 。
  • 💥關鍵特性（無 Tag）：它沒有標籤（It has no tags），因此僅能用來作為降低分支延遲的「暗示（Hint）」。
  • 運作流程：管線會假設這個 Hint 是正確的並順著該預測方向開始抓取指令（Fetching begins） ；如果最後發現暗示錯誤，該預測位元就會被翻轉（Inverted）並存回 。

📄 1-Bit 動態預測方案的極限 (Slide 45)

  • 更新條件：1-bit 預測器只有在發生預測錯誤（Mispredict）時才會更新其預測位元 。
  • 💥 經典打臉範例：考慮一個迴圈分支，其真實行為是連續 9 次 Taken，第 10 次為 Not Taken（跳出迴圈） 。
    • 雖然這個分支有 90% 的時間都在 Taken，但 1-bit 預測器在該分支留在緩衝區的假設下，正確率僅有 80%（8 次正確，2 次錯誤） 。
    • 原因：它會在迴圈首尾（第 10 次跳出時、以及下一輪重新進入迴圈時）各猜錯一次 。

📄 2-Bit 動態預測方案與狀態機 (Slides 46 - 48)

  • 判斷準則 (Slide 47)：使用一個 n-bit 計數器，當計數器的值大於或等於其最大值的一半時，分支預測為 Taken；否則預測為 Untaken 。
  • 加減更新：在 2-bit 方案中，分支若為 Taken 計數器就遞增，若為 Untaken 則遞減 。
  • 狀態機四種狀態 (Slide 46，如上圖)：
    • 11：Strongly Predict Taken
    • 10：Weakly Predict Taken
    • 01：Weakly Predict Not Taken
    • 00：Strongly Predict Not Taken
    • 好處：必須連續猜錯兩次才會徹底翻轉預測方向，解決了 1-bit 在迴圈邊界連續犯錯的問題 。
  • 硬體代價：由於 2-bit 方案更新預測位元的頻率高於 1-bit，且每個週期都需要讀取，因此 2-bit 預測器通常需要同時配備一個讀取存取埠（Read port）與寫入存取埠（Write port）。

📄 2-Bit 預測器的物理上限 (Slides 48 - 50)

  • 緩衝區容量上限 (Slide 48)：數據顯示，分支預測緩衝區的命中率（Hit rate）並不是限制精準度的關鍵因素 。如果完全不改變預測器的底層結構，單純增加每個預測器的 bit 數，對提升準確率幾乎沒有任何影響 。

🛑 3. 關聯式分支預測器 (Correlating / Two-Level Predictors) (Slides 51 - 59)

📄 跨指令關聯性代碼實例 (Slide 51)

投影片給出一段經典的 C 語言邏輯判斷，用以說明分支指令之間的相互牽引關係 ：

if (aa == 2) aa = 0;   // 分支 b1
if (bb == 2) bb = 0;   // 分支 b2
if (aa != bb) { ... }  // 分支 b3

對應的 MIPS 匯編代碼如下 ：

L1: DSUBUI R3, R1, #2      ; 測試 aa (R1) 是否等於 2
    BNEZ   R3, L1          ; 【分支 b1】
    DADD   R1, R0, R0      ; aa = 0
L2: DSUBUI R3, R2, #2      ; 測試 bb (R2) 是否等於 2
    BNEZ   R3, L2          ; 【分支 b2】
    DADD   R2, R0, R0      ; bb = 0
    DSUBU  R3, R1, R2      ; R3 = aa - bb
    BEQZ   R3, L3          ; 【分支 b3】

📄 關聯式預測器(Correlating Branch Predictors)的核心運作 (Slide 52 - 54)

  • 核心原理：分支 $b3$ 的最終行為與前面的分支 $b1$ 和 $b2$ 的行為高度相關（Correlated） 。傳統只看單一分支歷史的預測器永遠無法捕捉到這種行為 。
  • 定義：會利用其他分支的行為來為自己做出預測的預測器，就稱為關聯式預測器（Correlating Predictors）或兩級預測器（Two-Level Predictors） 。
  • 低成本的硬體實現：利用一個簡單的 $m$-bit 位移暫存器（Shift register） 來記錄全 CPU 最近發生的 $m$ 個分支各自是 Taken (1) 還是 Not Taken (0)，這被稱為全域歷史（Global history） 。

📄 關聯式預測器(Correlating Branch Predictors)查表機制詳解：以 (2,2) 預測器為例 (Slide 55)

  • 橫列選取（Row）：截取當前 Branch address（分支指令位址）的低位元（例如 4 個 bit），經由解碼後決定要查矩陣的哪一個橫列 。
  • 直欄選取（Column）：直接看當下的 2-bit 全域分支歷史（2-bit global branch history），這 4 種狀態（00, 01, 10, 11）決定了你要看矩陣的哪一個直欄 。
  • 小格子內存什麼（XX）：行列交叉選中的那個小格子 "XX"，內部儲存的就是一個 2-bit 的飽和計數器數值（00, 01, 10, 11） 。
  • 最終輸出：讀出該格子內的 2-bit 值後，大於等於 2 就輸出 Taken，小於 2 就輸出 Not Taken，這就是最終的 XX prediction 。

  • 關鍵數據對比：看最難對付的 eqntott 程式 。

🛑 4. 錦標賽預測器 (Tournament Predictors) (Slides 60 - 64)

📄 為什麼需要Tournament Predictors？ (Slide 60)

  • 關聯預測器的局限：關聯式預測器因為只看全域歷史（Global information），在處理某些非常重要的獨立區域分支時反而會失敗 。
  • 集成哲學：錦標賽預測器會同時內建多個預測器，通常包含一個基於全域資訊（Global）的預測器與一個基於區域歷史（Local）的預測器，並使用一個選擇器（Selector）來結合它們 。
  • 效能優勢：它在中等容量大小（8Kb - 32Kb）下能展現出更完美的準確度，並且能非常有效地運用大量的預測 bit 空間 。

📄 選擇器狀態機運作機制 (Slide 61 - 62)

  • 選擇器本質：現行的錦標賽預測器在每個分支 Entry 上使用一個 2-bit 飽和計數器（Saturating counter） 來作為挑選預測器的依據 。
  • 狀態機四大狀態 (Slide 62)：
    • 左半邊兩個狀態：Use predictor 1 。
    • 右半邊兩個狀態：Use predictor 2 。
  • 💥 狀態轉移與分數解讀：箭頭上的標記格式為 (預測器 1 對錯 / 預測器 2 對錯)，其中 1 代表猜對，0 代表猜錯 。
    • 看到 1/0：預測器 1 猜對、預測器 2 猜錯 $\rightarrow$ 狀態機向左滑（更信任預測器 1） 。
    • 看到 0/1：預測器 1 猜錯、預測器 2 猜對 $\rightarrow$ 狀態機向右滑（更信任預測器 2） 。
    • 看到 0/0 或 1/1：兩者分不出高下 $\rightarrow$ 狀態機留在原地保持不變 。
  • 業界頂級實例：著名的 Alpha 21264 處理器 分支預測就是採用這種精妙的錦標賽狀態機架構 。

📄 Tournament Predictors的實戰碩果 (Slide 63 - 64)

  • 智慧切換 (Slide 63)：以 Alpha 21264 跑 SPEC89 為例（縱軸代表「採用 Local 預測器做決定的比例」）：
    • 跑極度規律的科學計算 matrix300 時，選擇器 100% 全程聽從 Local 預測器的決定 。
    • 跑混亂複雜的 eqntott 時，選擇器只有 37% 聽 Local 的，剩下的 63% 自動切換改去聽 Global 的話 。
    • 這完美體現了它為正確的分支挑選正確預測器的強大能力 。
  • 終極對決曲線 (Slide 64)：在相同的總硬體預測器容量限制下（Total predictor size），由上到下的性能表現（錯誤率由高到低）排序為：傳統 Local 2-bit 預測器 → Correlating 關聯式預測器 → Tournament 錦標賽預測器（錯誤率最低、表現最完美） 。

🛑 Section 1: 動態排程與亂序執行的物理限制 (Slides 65 - 70)

1. 為什麼要發明動態排程技術？ (Slide 65)

  • 硬體核心定義：動態排程是指由硬體在執行期（Runtime）動態重新排列指令的執行順序，以達到減少管線停頓（Stalls）的目的，同時必須完美維持程式原本的資料流（Data flow）與例外行為（Exception behavior） 。
  • 兩大絕對優勢：
    1. 能在執行期即時處理一些編譯時期無法預知的卡頓情況（例如最經典的快取缺失 Cache Miss） 。
    2. 能顯著簡化編譯器的設計複雜度 。
  • 代價：會導致硬體複雜度與晶片電路成本的大幅飆升 。

2. 依序發射（In-Order Issue）的物理瓶頸 (Slide 66, 67)

  • 傳統管線採用依序發射，如果前方的指令因為相依性或硬體卡住而 Stalled，管線會被當場堵死，導致後方完全不相干、原本可立刻執行的指令通通被強迫扣留，造成多個實體運算單元（Functional Units）陷入閒置（Idle） 。
  • 投影片經典範例（Slide 67）：
    DIV.D   F0, F2, F4   ; 浮點數除法：運算極慢 [cite: 756, 757]
ADD.D   F10, F0, F8  ; 真資料依賴（RAW）：必須等待 F0 [cite: 758, 759, 762]
SUB.D   F12, F8, F14 ; 獨立的減法 [cite: 760, 761, 763]

    在傳統管線中，因為 ADD.D 被 DIV.D 卡住，完全獨立的 SUB.D 也會被硬生生堵死在後半段，形成嚴重的效能瓶頸 。

3. 亂序完成（Out-of-Order Completion）引發的併發症 (Slide 68 - 70)

  • 名稱依賴危險：允許亂序執行（Out-of-order execution）必然伴隨著亂序完成 。這時，跨指令間的假依賴（名稱依賴）會直接被觸發為硬體 Hazards ：
    • WAR Hazard (反依賴)：後面的指令改寫速度太快，提早改寫了前方指令正準備讀取的暫存器舊值 。
    • WAW Hazard (輸出依賴)：多道指令同時搶著改寫同一個目的地暫存器，若完成順序顛倒，會導致最終留下來的暫存器數值發生錯誤 。
  • 管線 ID 階段的一分為二（Slide 69）：
    為了解鎖亂序，硬體把傳統管線的 ID 階段（指令解碼）切割成兩個實體步驟：
    1. Issue（發射）：解碼指令，並嚴格檢查有沒有結構阻礙（Structural Hazards） 。
    1. Read operands（讀取運算元）：一邊排隊，等到確定沒有任何真資料危險（RAW）後，才真正把參數值讀進去 。
  • 非精確例外 (Imprecise Exceptions)（Slide 70）：
    如果後面的指令提早跑完並修改了暫存器，而排在前面的指令在此時突然發生錯誤（Exception），這會導致 CPU 的暫存器狀態無法完美倒帶回 sequential 順序，使得作業系統無法重新啟動（Restart）程式。

    • 由 FP registers 垂直向下指引到保留站的輸入端，負責遞送已就緒的乾淨暫存器數值。
  • Operation bus（操作碼匯流排）
    • 從 Instruction queue 水平橫向拉到保留站與 Load-store 區，負責指派當前的操作碼（如 ADD.D）。
  • CDB（通用資料總線）的注入端與反向回餵端
    • 注入端（箭頭向下）：FP adders 的輸出端、FP multipliers 的輸出端、以及 Memory unit（記憶體單元）的 Load data 輸出端，皆有實體線路垂直向下注入 CDB。
    • 回餵端（箭頭向上）：CDB 的實體連線會向上逆流，同時連回最上方的 FP registers、中央的 Reservation stations，以及左側的 Store buffers。
🚦 Section 4: Tomasulo 三大階段與保留站分配邏輯 (Slides 76 - 82)

1. 🌟 【你特別提問的大考點】保留站「座位名稱（Name）」到底是怎麼指派的？

  • 鐵律：Load1、Add1、Mult1 這些名字是硬體在出廠時就做死的實體座位代號（Tags） 。
  • 分配機制（動態遞增佔位）：當指令從 Instruction Queue 依序發射出來時，硬體會看它的操作碼（Op），去對應的保留站類型中，尋找目前 Busy = no（沒人坐）且座位編號最小的那一個 Entry，強勢將其霸佔並改標為 Busy = yes 。
  • 舉例：若連續進來兩個加法指令，第一個會自動坐進 Add1；第二個進來時，發現 Add1 正忙，便會自動坐進下一個空位 Add2 。

2. 三大管線階段硬體運作鐵律 (Slides 76 - 81)

階段一：Issue (發射) (Slide 77)

  • 動作：從指令佇列頭部抓取指令 。
  • 條件：對應類型的保留站必須有空位（Empty slot） [cite: 842]。
    • 有空位→ 指令發射。如果來源暫存器的 Qi 欄位為空白，代表數值現成，直接把值複製進保留站的 Vj/Vk 欄位；如果來源暫存器的 Qi 寫著別人的名字，代表資料還在卡關，則在保留站的 Qj/Qk 欄位填入那個人（保留站）的 Tag，進入等待 。
    • 同步改寫暫存器擁有權：強行將目的地暫存器的 Qi 欄位修改改寫為當前這個保留站的名字（完成 Register Renaming，藉此消滅 WAW）。
    • 沒有空位→觸發 Structural Hazard，指令直接卡在佇列頭部，管線整排 Stall 。

階段二：Execute (執行) (Slide 78, 79)

  • 動作：保留站張大眼睛盯著最底部的 CDB 巴士 [cite: 849][cite_start]。一聽到自己 Qj/Qk 正在等待的 Tag 廣播，立刻把對應的實體數值抓進 Vj/Vk 欄位中，並把 Qj/Qk 清空為 0 。
  • 條件：當所有的來源運算元都完全到齊（即 Qj 和 Qk 通通變為 0/空白，Vj 和 Vk 皆拿到 Value），運算單元立刻開始正式執行運算（藉此避開 RAW 阻礙）。

階段三：Write Result (寫回結果) (Slide 81)

  • 條件：運算單元執行完畢，且順利搶奪到了 CDB 總線的廣播權。
  • 動作：將算好的數值，連同自己這個保留站的座位 Tag，同時廣播到 CDB 上。所有在排隊等這個 Tag 的保留站欄位（Qj/Qk）以及暫存器 Qi 欄位，會同步把數值吸進去，並把 Tag 清空。隨後，當前這個保留站功成身退，座位變回空閒狀態（Busy = no） 。

🛑 Section 6: 記憶體衝突與 Tomasulo 的三大核心組件 (Slides 92 - 94)

1. 記憶體指令交錯發射（Interchanging）的 Hazard 鐵律 (Slide 92)

如果我們在管線中試圖打亂 Load 和 Store 指令的先後順序，如果它們存取到同一個記憶體位址（Same address），會引發嚴重的記憶體 Hazards ：

  1. Load 挪到 Store 後面 → 原本應該先讀取舊值，結果順序對調變成先寫入再讀取 → 觸發 WAR Hazard 。
  1. Store 挪到 Load 後面 → 後面的 Load 會讀到錯誤的舊資料 → 觸發 RAW Hazard 。
  1. 兩個 Store 互相調換順序 → 記憶體最終留下的數值會顛倒錯亂 → 觸發 WAW Hazard 。
  1. 硬體安全防護手段：為了解決這個問題，Tomasulo 規定：所有的 Load 和 Store 指令，其「有效位址計算（Effective address calculation）」這一步，必須嚴格按照原始的程式順序（Program order）來依序進行。只要位址確定不撞車，後續的讀寫就能放心地亂序排程了 。

🛑 Section 1: 硬體投機執行 (Hardware-Based Speculation) 核心概念 (Slides 95 - 97)

1. 為什麼要引入投機執行？

  • 控制相依性的沉重負擔：隨著我們試圖壓榨出更高的指令層級平行度（ILP），控制相依性（Control dependences，即分支指令）迅速成為限制 CPU 效能發揮的最主要枷鎖 。
  • 打破限制的手段：克服控制相依性的核心方法，就是直接對分支指令的最終結果進行猜測/投機（Speculating），並在分支結果尚未真正算出來前，就假裝我們的猜測是百分之百正確的，直接往下執行程式碼 。
  • 技術的延伸：硬體投機執行在本質上，就是將動態分支預測、動態指令排程（Tomasulo）的精髓進行了更進階的硬體優化與擴充 。

2. 投機執行的四大核心支柱 (Slide 96)

  1. 動態分支預測 (Dynamic branch prediction)：負責在管線前端精準挑選出「接下來應該執行哪一條路徑的指令」 。
  1. 投機執行 (Speculation)：允許指令在引發控制相依性的分支尚未解決（Resolved）前，就提前重疊執行 。
  1. 動態指令排程 (Dynamic scheduling)：負責處理並編排跨越不同基本區塊（Basic blocks）之間的指令組合與執行順序 。
  1. 減少分支停頓：分支預測能消滅絕大部分由分支指令帶來的直接管線停頓（Direct stalls） 。

3. 多重發射處理器的剛性需求 (Slide 97)

  • 對於一個單週期執行多道指令（Multiple instructions per clock）的寬發射處理器而言，單純只做到分支預測準確是不夠的 。
  • 為了維持最大化效能，寬發射處理器可能需要在每個時鐘週期都執行並判定一道分支指令 。因此，我們必須徹底跨越控制相依性帶來的物理邊界 。
🛡️ Section 2: 核心硬體元件——重新排序緩衝區 (Reorder Buffer, ROB) (Slides 98 - 100)

1. 擴充 Tomasulo 的核心哲學：分離結果與更新 (Slide 98)

  • 為了支援投機執行，我們必須將 Tomasulo 演算法的硬體進行升級 ：「必須將指令之間的結果前饋前饋（Bypassing），與指令真正的完成更新（Completion/Update）進行徹底的實體分離！」
  • 留校察看機制：只要指令仍處於「投機狀態（Speculative）」，它算好的結果就只被允許在指令間前饋傳播並存入暫存區，絕對不准更新處理器的真實狀態（暫存器與記憶體），直到該指令被證實不再具有投機性為止 。

2. 什麼是 Reorder Buffer (ROB)？ (Slide 99)

  • 硬體角色：為了在管線中加入這個安全更新的「提交階段（Commit phase）」，硬體引入了 ROB（重新排序緩衝區） 。
  • 前饋來源：ROB 負責在可能屬於投機執行的指令之間，安全地傳遞並前饋運算結果 。
  • 緩衝時間點：它負責保存指令的結果，時間橫跨「指令操作完成（Completes）」到「指令正式安全提交（Commits）」 之間的這段等待期 。
  • 與 Tomasulo 的共通點：ROB 與 Tomasulo 的保留站一樣，在執行期都能作為後續指令尋找來源運算元（Operands）的現成供應源 。

3. 💥 【期中考必背觀念】Tomasulo 與 ROB 架構的關鍵本質差異 (Slide 100)

  • 在純 Tomasulo 演算法中：一旦指令將結果算好並寫回（Write Result），後續發射的指令就能直接在實體暫存器檔案（Register File）中找到這個最新的結果 。
  • 在 speculative 投機處理器中：指令算完寫回時，實體暫存器檔案絕對不會被更新，數值會一直被扣留在 ROB 中，直到該指令正式走到頭部提交（Commits）時，暫存器才會被真正改寫！

4. ROB 的 4 大核心硬體欄位 (Slide 100)

ROB 是一個實體環形佇列（Circular Queue），每一個 Entry 均包含以下 4 個必備欄位：

  1. Instruction type（指令類型）：標記該指令是 Branch（分支）、Store（記憶體寫入）還是 Register operation（暫存器寫入運算） 。
  1. Destination field（目的地欄位）：記錄結果最終要寫入哪一個實體暫存器編號（如 F0），或是對 Store 指令而言的記憶體寫入位址 。
  1. Value field（數值欄位）：用來暫時存放運算單元已經算好、發射在 CDB 上，但尚未獲准真正提交的運算結果數值 。
  1. Ready field（就緒欄位）：標記該指令是否已經在運算單元中執行完畢並將結果送回 ROB（Yes/No） 。

🚦 Section 4: 加入 ROB 後全新的四階段管線運作鐵律 (Slides 102 - 109)

加入 ROB 後，管線正式由三階段擴充為四個實體階段（Four Pipeline Stages） ：

階段一：Issue (發射) (Slide 103)

  • 動作：從指令佇列（Instruction Queue）的頭部抓取一道指令 。
  • 強制觸發條件：同類型型態的保留站（Reservation Station）必須有空位，而且 ROB 內部也必須有空的暫存格（Empty slot in the ROB） 才可以發射 。若任一硬體資源不夠，即觸發結構阻礙（Structural Hazard）卡住 。
  • 硬體分配動作：
    1. 派發保留站座位，並將控制 Entry 標記為使用中（In use） 。
    1. 如果來源運算元在實體暫存器或 ROB 中已經算好現成可用，立刻複製到保留站 。
    1. 💥 【最核心暫存器重新命名改變】：硬體會將這道指令被指派的 ROB Entry 編號（Tag）同步發送給保留站。並且，目的地暫存器的狀態 Qi 不再填寫保留站名字，一律強勢修改填寫為該指令分配到的「ROB Entry 編號」！ 後續新指令如果需要這個資料，就去這個 ROB 編號拿 。

階段二：Execute (執行) (Slides 104 - 106)

  • 動作：本階段旨在全面檢查並防範 RAW 危害（真資料相依性） 。
  • 運作機制：如果來源運算元尚未就緒，保留站張大眼睛死死監控 CDB 巴士 。當兩道來源運算元在保留站中通通就緒（Qj/Qk 皆清空為 0 拿到實體 Value），運算單元立刻動手執行操作 。
  • 記憶體指令的特殊兩步執行 (Slide 105, 106)：
    • 第一步：計算有效位址（Effective address），並將位址登載入 Load 或 Store 緩衝區中 。
    • 第二步（Load）：只要記憶體單元（Memory Unit）一空閒，Load 立刻出發去記憶體抓資料 。
    • 第二步（Store）：在 Execute 階段，Store 只需要確保其基底整數暫存器（Base register）就緒以供計算位址即可，因為它此時在管線中只需要完成有效位址計算 。
    • 順序限制：Load 和 Store 指令其有效位址計算，必須嚴格維持程式順序（Program order），這能完美防範經由記憶體爆發的 Hazards 。

階段三：Write Result (寫回結果) (Slides 107 - 108)

  • 觸發條件：運算單元執行完畢，且順利搶奪到了 CDB 總線的廣播控制權。
  • 硬體分配動作：
    1. 將計算出來的結果數值，連同發射時指派給它的 ROB 編號（Tag），大聲廣播發射到 CDB 上 。
    1. 數值順著 CDB 灌入 ROB 對應編號的 Value 欄位，並同步送給全天下正在排隊等這個 ROB Tag 的保留站 。
    1. 將該指令佔用的保留站正式釋放，標記為現成可用（Available） 。
  • ⚠️ 【Store 指令的特殊 Write Result 動作】(Slide 108)：
    • 如果 Store 指令要寫入記憶體的「數值」此時在保留站中已經算好了，該數值會被直接寫入該 Store 指令對應的 ROB 條目中的 Value 欄位中 （此處依然不准改寫記憶體！）。
    • 如果數值還沒好，Store 的 ROB Entry 會持續死死監控 CDB 廣播，直到數值送達並更新其 ROB 的 Value 欄位為止 。
階段四：Commit (提交 / 畢業) (Slide 109)
  • 核心機制：這是 ROB 的靈魂。所有的指令必須嚴格按照原始的程式順序（In-order），在 ROB 的頭部（Head）排隊依序提交畢業。
  • 硬體依據 3 大情境動作：
    1. 正常運算指令（Normal Commit）：當該指令到達 ROB 頭部，且其 Ready 欄位被標記為 yes（代表 Write Result 已完成）。硬體會正式將 ROB Value 欄位中的數值寫入實體暫存器檔案（Register File），該指令正式從 ROB 畢業清空 。
    1. Store 指令提交（Store Commit）：與正常指令相似，但最大的區別在於：此時數值不是寫入暫存器，而是正式大筆一揮寫入實體記憶體單元（Update Memory） ！
    1. Branch 指令提交（猜錯分支時）：當一條分支指令到達 ROB 頭部時，硬體會比對真實結果與當初的預測。如果發現當初分支預測錯誤（Incorrect prediction），這強力代表當前 ROB 裡面排在該分支後面的所有投機執行指令通通作廢（Speculation was wrong） ！硬體會啟動大清空機制（下一節詳細說明）。

1. Store 指令在 Tomasulo 與 Speculative ROB 架構中的處理差異 (Slide 114)

  • 在傳統 Tomasulo 演算法中：Store 指令只要有效位址計算完畢，且要寫入記憶體的資料數值在保留站中就緒，就能在 Write Result 階段直接更新改寫記憶體 。
  • 在 speculative 投機處理器中：Store 指令算好位址與數據後，只能先存放在 ROB 的 Value 欄位中 。它只有當自己到達 ROB 的頭部（Head）時，才被允許真正寫入更新記憶體 。這條底線確保了在指令被證實「不再具有投機性」前，記憶體絕對不會被錯誤污染 。

2. ROB 處理分支預測錯誤（Misprediction）的復原機制 (Slide 115)

  • 透過 ROB，處理器可以非常輕鬆且完美地復原猜錯分支後的系統狀態 ：
    1. 當一條預測錯誤的分支指令平安到達 ROB 的頭部（Head）時，硬體正式確認投機執行失敗 。
    1. 大清空（Flush）：硬體直接清空並廢除在 ROB 中排在該分支後面的所有指令 Entries 。
    1. 允許排在該分支之前的合法指令繼續在 ROB 中依序完成 Commit 。
    1. 將 PC 指標重新導向至正確的分支路徑上，重新開始抓取指令（Restart fetch） 。

3. ROB 處理精確例外（Precise Exception）的底線邏輯 (Slide 116)

  • 留校察看：如果一條投機執行的指令在 Execute 階段引發了硬體例外（Exception），處理器絕對不會立刻報錯中斷，而是將例外狀態默默記錄在該指令對應的 ROB 條目中 。
  • 無痛清除：如果後續發現分支預測失敗，這條不該被執行的指令會直接被 Flush 清空，這個例外也會跟著消失，完美避免了誤報 。
  • 正式觸發：只有當這條帶有例外標記的指令順利熬到 ROB 的頭部準備 Commit 的那一瞬間，CPU 才會正式觸發例外處理機制 。這完美達成了精確例外模型 。

4. 如何利用 ROB 消除記憶體相依性 Hazard？ (Slide 117, 118)

  • 消除 WAW 與 WAR 記憶體危險 (Slide 117)：因為實體記憶體的寫入改寫被嚴格限制在 Commit 階段，且 Commit 是嚴格依序（In-order）發生的 。因此，在 Store 畢業前，絕對不可能有任何後方的讀寫指令能提早越界改寫記憶體，Hazards 直接在物理層面被消滅 。
  • 維護 RAW 記憶體真依賴 (Slide 118)：硬體實施了兩大剛性限制：
    1. 位址比對攔截：如果管線中有一道剛進來的 Load 指令，發現目前在 ROB 裡排在它前面的某個 Store 指令，其目的地欄位（Destination / 即有效位址 A 欄位）與當前這道 Load 準備存取的位址完全吻合，硬體強制攔截並禁止這道 Load 啟動其第二步的記憶體存取執行，直到 Store 的 Value 到齊為止 。
    1. 依序計算位址：Load 與 Store 計算有效位址的動作，必須嚴格維持程式順序（Program order） 。

2. 靜態超純量 vs. VLIW 的本質差異 (Slide 120 - 123)

  • Superscalar（超純量）：具備「動態發射能力（Dynamic issue capability）」 。硬體自己決定這個週期要同時抓取並發射 0 到 8 道指令，並在發射時即時檢查 Pipeline Hazards 。
  • VLIW（超長指令字）：具備「靜態發射能力（Static issue capability）」 。硬體不做任何防護，把「同時發射指令」的重責大任完全推給編譯器（Compiler） 。
    • 指令打包（Issue Packet）：編譯器在編譯時，就把 5 道互不相干的操作打包成一條超級長的指令（長度高達 112 到 168 bits） 。例如：1 個整數/分支操作 + 2 個浮點數操作 + 2 個記憶體讀寫 。
    • 優點：硬體不需要複雜的 Hazard 檢查電路，結構極其簡單、省電 。
    • 缺點：如果程式碼找不到足夠多的獨立指令，格子沒填滿，編譯器就必須在長指令裡塞滿無用的 NOP（空指令），浪費記憶體空間（Wasted bits） 。另外，一個是為了使平行度增加，VLIW 的編譯器會做激進的loop unroll，code size 會增加、因為 VILW 是在一個 instruction 完成多種操作，所以編譯器要考量到相依性問題，所以會留有大量的 NOP、每個 ISA 的實現不同，因此 CPU 本身的 VLIW 也會不同，需考量相依性問題。

🛑 Section 3: VLIW 的現實技術與物流瓶頸 (Slides 127 - 131)

雖然 VLIW 架構簡單，但在商業市場上卻遇到了致命障礙（期中考必考簡答）：

  1. Lock-Step Synchronization（同步鎖定限制）(Slide 128)：
    VLIW 內部的所有功能單元必須保持嚴格同步 。只要其中一個管道（例如記憶體管道發生了 Cache Miss 沒抓到資料），整個處理器、所有的運算單元通通都要一起被強迫暫停（Stall） ！這在記憶體延遲不固定的現代系統中代價太高了 。
  1. Binary Code Compatibility Crisis（二進位代碼相容性危機）(Slide 129)：
    VLIW 的程式碼是編譯器根據該晶片「特定的運算管道數量、特定的硬體延遲（Latencies）」去精準填格子的 。
    • 悲劇：如果下一代晶片把加法器從 2 個改成 3 個，或者優化了電路讓延遲縮短了 1 週期，上一代的舊程式就完全沒辦法在新晶片上執行，必須全部重新編譯（Lack of compatibility） 。

💡 核心解碼心法（考試怎麼推導出來的？）：

  • 為什麼 DADDIU R2 在 Cycle 5 才 Execute？ 因為它資料依賴於 LW 。LW 在 Cycle 4 才 Write CDB 把 R2 吐出來，所以它最快只能在 Cycle 5 動手執行 。
  • 為什麼 SW 在 Cycle 7 才做 Memory Access？ SW 的有效位址在 Cycle 3 就用基底整數暫存器算好了（Execute = 3） 。但它要存入記憶體的資料是 R2 的最新值，這筆資料是由 DADDIU R2 掌管的，而該指令在 Cycle 6 才 Write CDB。因此，SW 必須在 Cycle 7 才能真正去寫入記憶體 。
  • 為什麼第二圈的 LW 到了 Cycle 8 才能 Execute？ 這就是無投機的致命傷！第一圈的 BNE 需要用到 DADDIU R2 算出來的 R2，而 R2 在 Cycle 6 才廣播。所以 BNE 必須等到 Cycle 7 才能完成條件評估（Execute = 7） 。

1. 分支目標緩衝區 (Branch-Target Buffer, BTB) (Slide 138)

前面我們學過傳統的 BPB（分支預測緩衝區），它必須等到 ID（解碼）階段 才能看出這是一條分支指令並進行猜測 。
為了解決這個問題，科學家發明了 BTB ：

  • 運作原理：它直接是一塊用當前 PC 位址去 Look up 的 Cache 。
  • 震撼特點：它在 IF（抓取指令）階段 就能發揮作用 。只要當前抓取指令的 PC 撞中了 BTB 內的紀錄（Hit），BTB 就會在同一個週期直接把「預測的跳轉目標 PC 位址（Predicted PC）」吐給程式計數器 。
  • 帶來的效益：如果預測成功（Taken 且猜對），處理器可以達到 0 週期跳轉懲罰（Zero-cycle branch penalty），管線完全不需要任何卡頓 。

🛑 Section 7: 進階預測優化技術名詞快譯 (Slides 144 - 154)

這章的最後幾頁介紹了許多現代高端微架構的進階優化名詞，通常會出在是非題、選擇題或名詞解釋中：

  • Branch Folding（分支摺疊）(Slide 144)：
    在 BTB 裡除了存預測的 PC，乾脆直接把目標位址的那道「實體指令（Actual target instruction）」也一起緩衝進去 。這樣一來，遇到無條件跳轉時，連抓指令的動作都省了，達成真正的 0 週期無條件分支 。
  • Return Address Predictors（返回位址預測器）(Slide 145)：
    專門用來對付副程式呼叫的 return 指令 。因為同一個函數可能在程式中不同的地方被呼叫，返回位址（Indirect Jumps）會變來變去，傳統 2-bit 預測器會猜得很慘 。
    • 硬體解法：晶片內建一個小型的硬體堆疊（Return Address Stack, RAS），遇到 call 就把當前 PC 壓入（Push），遇到 return 就直接從彈出（Pop）位址進行預測，準確度極高 。
  • Value Prediction（數值預測）(Slide 153)：
    這是一個非常前衛的思想：資料依賴（RAW）之所以會卡住，是因為後面在等前面算出來的「數值」 。那我們能不能連數值都直接用猜的 ？如果猜中了（通常用於常數池載入或規律遞增的變數），後續的一整串真資料相依指令就能提早平行執行 。直接猜測指令會算出來的「數值」，適合用於極少改變的變數（如常數載入），藉此提早解除資料依賴 。
  • Address Aliasing Prediction（記憶體別名預測）(Slide 154)：
    在有效位址還沒算出來之前，提早預測這道 Load 與前面的 Store 到底是不是存取同一個記憶體格子，藉此決定 Load 能不能大膽地提早跑 。
  • Thread-Level Parallelism (TLP) 與 SMT：當單一執行緒的 ILP 被榨乾時，利用同步多執行緒 (Simultaneous Multithreading, SMT) 讓多個獨立程式在同一個週期內共用硬體運算單元，大幅提升處理器吞吐量 。

📝 考前終極驗收：20 題是非題附詳細解析

這 20 題涵蓋了第三章中後段的所有重點坑點：
1. 在 VLIW（超長指令字）處理器中，硬體會負責在指令發射時（Issue time）動態檢查並解決資料依賴與管線危險。

答案：X

詳解：VLIW（超長指令字）的核心哲學就是「硬體極簡化」。危害偵測與指令排程的責任完全落在軟體（編譯器）身上 。硬體不會在執行期動態檢查衝突，如果編譯器排錯了，程式就會直接崩潰。


2. 靜態超純量（Statically-scheduled superscalar）處理器與 VLIW 處理器一樣，每次時鐘週期都必須發射固定數量的指令，不能動態改變發射數量。

答案：X

詳解：靜態超純量（Superscalar）雖然指令是依序發射，但硬體具備動態發射能力（Dynamic issue capability），可以根據當下資源狀況決定發射 0 到 8 道指令 。而 VLIW 才是每週期固定吞入編譯器設定好的靜態指令包 。


3. BTB（分支目標緩衝區）的作用階段是在管線的 IF（指令獲取）階段，這比傳統的 BPB（分支預測緩衝區）在 ID 階段得知預測結果還要早一個週期。

答案：O

詳解：完全正確。傳統的 BPB 需要先抓指令出來解碼（ID 階段）才知道它是 Branch ；而 BTB 在 IF 階段直接拿 PC 去查表，命中就能直接替換下一週期的 PC，提早了一個週期發揮作用 。


4. 假設一個分支指令沒有在 BTB 中命中（預測不跳轉），但最後它的真實行為是「跳轉（Taken）」，此時管線不需要付出任何懲罰週期（0 cycle penalty）。

答案：X

詳解：如果在 BTB 沒命中，代表硬體「預測不跳轉（繼續往下走）」。但如果真實行為是「跳轉（Taken）」，代表管線抓錯了指令，必須在解碼或執行後清空管線，這會產生 2 個週期的分支懲罰（2-cycle penalty） 。


5. 「分支摺疊（Branch Folding）」技術是將實際的目標指令直接儲存在 BTB 中，這有時可以實現零週期的無條件分支甚至條件分支。

答案：O

詳解：正確。這就是 Branch Folding（分支摺疊）技術。透過把目標指令快取在 BTB 中，當命中無條件跳轉時，連去記憶體抓目標指令的時間都省了，達成真正的零週期跳轉 。


6. 返回位址預測器（Return Address Predictors）使用一個小型堆疊（Stack）結構，主要用來解決因為副程式從多個不同地方被呼叫，導致返回位址難以預測的問題。

答案：O

詳解：正確。副程式的 return 位址會因為呼叫來源不同而動態改變（Indirect jumps）。使用硬體 Stack（RAS）在 call 時壓入、return 時彈出，可以極其精準地預測返回位址 。


7. 即使我們打造出一顆「完美的處理器」（無限暫存器、完美分支預測、完美快取），程式的 ILP 依然會受到「Window Size（指令視窗大小）」的物理比較次數限制。

答案：O

詳解：完全正確。要在硬體中動態找出可以平行的指令，必須進行相依性比較。比較的次數與 Window Size 呈平方級數成長，硬體的比較器數量（如上千個 comparator）是真實世界的物理極限 。


8. 在動態排程處理器中，如果 Load 與 Store 指令存取同一個記憶體位址，只要它們的運算元都就緒了，硬體就可以任意調換它們的執行順序。

答案：X

詳解：絕對不行！如果 Load 和 Store 存取「同一個位址」，任意調換順序會引發致命的 WAR、RAW 或 WAW 記憶體危險 。Tomasulo 規定，所有記憶體指令的「有效位址計算」必須嚴格按照程式順序進行，確定不衝突後才能亂序讀寫 。


9. 在支援硬體投機執行（Speculation）且配備 ROB 的架構中，Store 指令可以在 Write Result 階段直接將數據寫入實體記憶體中，以節省時間。

答案：X

詳解：經典陷阱題！在有 ROB 的投機架構中，Store 算好的數據只能先存放在 ROB 的 Value 欄位裡 。只有當 Store 走到 ROB 的頭部（Head）並進行 Commit（提交）時，才被允許真正更新實體記憶體，這是為了防止分支猜錯而誤改記憶體 。


10. ROB（重新排序緩衝區）架構能完美支援精確例外（Precise Exceptions），因為當指令在 Execute 階段發生例外時，CPU 會立刻中斷程式並進行例外處理。

答案：X

詳解：ROB 的精確例外機制是「延後觸發」。當指令在 Execute 階段發生例外時，CPU 不會立刻中斷，而是將例外記錄在 ROB 中 。只有當這條指令走到 ROB 的頭部（Head）且確定要 Commit 時，才會正式觸發例外 。


11. 在 (m,n) 關聯式分支預測器中，硬體是將「全域分支歷史（m bits）」與「分支指令位址的低位元」拼接起來，共同作為查表的索引（Index）。

詳解：正確。(m,n) 預測器將 m bit 的全域歷史與分支位址低位元拼接（Concatenation），共同作為預測緩衝區的查表索引，藉此捕捉跨指令的關聯性 。

13. 「數值預測（Value Prediction）」主要用來預測那些頻繁且隨機改變的運算結果，藉此大幅提升浮點數運算的效能。

答案：X

詳解：數值預測（Value Prediction）主要是針對那些極少改變（infrequently changed）的數值，例如常數池的載入，或者是固定模式遞增的計數器 。如果數值隨機頻繁改變，預測正確率太低，反而會因為猜錯而付出巨大的 Flush 代價 。


14. 錦標賽預測器（Tournament Predictor）內部擁有一個選擇器（Selector），這個選擇器也是一個 2-bit 飽和計數器，用來記錄哪個預測器最近猜得比較準。

答案：O

詳解：正確。錦標賽預測器通常結合 Local 與 Global 預測器，並使用一個 2-bit 飽和計數器作為 Selector。如果 Global 猜對 Local 猜錯，狀態機就會往 Global 傾斜，達成動態智慧切換 。


15. 細粒度多執行緒（Fine-Grained Multithreading）通常只在遇到昂貴的 Cache Miss（快取缺失）時才會進行執行緒的切換。

答案：X

詳解：相反了。細粒度（Fine-Grained）是「每個時鐘週期（every instruction）」都在輪流切換執行緒 。粗粒度（Coarse-Grained）才是遇到昂貴的 Stall（如 L2 Cache Miss）時才進行切換 。


16. SMT（同步多執行緒）結合了超純量與多執行緒技術，允許在同一個時鐘週期內，同時發射來自「不同執行緒」的指令。

答案：O

詳解：正確。這正是 SMT（Simultaneous Multithreading）的強大之處。它利用多重發射與動態排程的硬體資源，在同一個 Cycle 內混搭發射多個執行緒的指令，徹底利用閒置的運算單元 。


17. 在 VLIW 架構中，如果遇到某個功能單元卡住（Stall），硬體可以只暫停該功能單元，讓其他功能單元繼續執行以維持效能。

答案：X

詳解：這正是 VLIW 的致命傷：Lock-step synchronization（同步鎖定限制） 。因為 VLIW 的指令是靜態綁定在同一個大封包裡的，只要其中一個功能單元（如 Memory miss）卡住，整個處理器所有的功能單元都必須一起暫停（Stall） 。


18. 迴圈展開（Loop Unrolling）雖然可以減少迴圈的控制開銷（Overhead），但展開次數過多會導致暫存器壓力（Register Pressure）大增，反而可能降低效能。

答案：O

詳解：完全正確。迴圈展開會大幅減少分支與索引更新的 Overhead，但由於大量的迭代被拉平同時執行，會導致「同時存活的變數（Live values）」暴增 。如果超出實體暫存器數量，就會產生極大的暫存器壓力（Register Pressure），導致變數被迫溢出（Spill）到記憶體中，效能崩潰 。


19. 基於歷史剖析（Profile-Based）的靜態分支預測，其科學基礎在於多數分支指令的行為呈現「雙峰分布（Bimodally distributed）」，亦即傾向於一直跳轉或一直不跳轉。

答案：O

詳解：正確。基於剖析（Profile-Based）的靜態預測之所以準確，就是因為觀察到分支行為通常是「雙峰分布（Bimodally distributed）」的，習慣跳轉的分支未來有極高機率繼續跳轉 。


20. 當分支預測錯誤時，ROB 的復原機制是「清空（Flush）ROB 中排在該分支前面的所有指令」，並重新開始抓取。

答案：X

詳解：方向反了！當分支猜錯並在 ROB 頭部被發現時，硬體是清空 ROB 中排在該分支後面（After）的所有投機指令 ，因為它們是被錯誤路徑帶進來的。而排在該分支前面（Before）的指令是合法的，會被允許繼續執行並 Commit 。

🛑 Section 1: 烏托邦中的「完美處理器」 (Slides 155 - 158)

如果我們移除所有硬體限制，ILP（指令層級平行度）能衝到多高？科學家定義了一顆「完美處理器（Perfect Processor）」，它具備以下 5 大超能力：

  1. 完美的分支預測（Branch prediction）：永遠不會猜錯 if/else。
  1. 完美的跳轉預測（Jump prediction）：完全消滅控制阻礙（Control stalls）。
  1. 完美的記憶體去別名化（Memory-address alias analysis）：硬體能完美知道 Load 和 Store 是不是指向同一個位址，不會產生無謂的等待。
  1. 完美的快取（Perfect caches）：所有的 Load 和 Store 都能在 1 個時鐘週期（1 cycle） 內完成，0 延遲，沒有 Cache Miss。

  • 當實體暫存器用完時，管線就會因為 Structural Hazard 而停頓。

4. 不完美的記憶體去別名化 (Slides 174-178)

  • 在執行期（Runtime），硬體很難完美預知這道 Load 會不會跟前面的 Store 撞位址。
  • 如果採用保守的「檢查（Inspection）」機制，很多安全的 Load 也會被逼著排隊，導致 ILP 大幅下降。

🛑 Section 1: 從 ILP 邁向 TLP 的範式轉移 (Slides 193 - 197)

1. 為什麼單靠 ILP 走進了死胡同？

  • 隱性平行度的枯竭：ILP（指令層級平行度）是由硬體在背後偷偷尋找完全不相干的指令來重疊執行，對軟體工程師是完全透明（隱性）的。
  • 功耗牆與硬體開銷的代價：為了多壓榨出 10% 的單核效能，動態排程與投機執行的硬體複雜度（如比較器、ROB Entries）會呈二次方甚至指數級飆升，耗電量大增，效益極低。

2. 執行緒層級平行度 (Thread-Level Parallelism, TLP) 的崛起

  • 核心定義：TLP 是指在軟體架構中，將程式切分成多個獨立、可並行執行的執行緒（Threads）。這些執行緒通常是顯性（Explicit）的，由程式設計師或作業系統排程。
  • 物理優勢：多個執行緒之間的資料相依性極低（甚至完全獨立），因此硬體不需要大費周章去比對暫存器衝突，能以極低的硬體代價換取巨大的系統吞吐量 (Throughput)。
📊 Section 2: 【大考絕對核心】硬體多執行緒三大流派發射槽對抗 (Slides 198 - 204)

考試最喜歡考這三個流派的名詞解釋、優缺點對比，以及畫圖/填空題（分析四種空缺格子的行為）。

1. 細粒度多執行緒 (Fine-Grained Multithreading)

  • 運作機制：處理器在每一個時鐘週期（Every single cycle），都會利用輪詢（Round-Robin）的方式強迫切換到不同的執行緒來發射指令。
  • 優點：能完美隱藏所有短暫的管線停頓（如 1 週期的資料依賴、分支延遲）。如果執行緒 A 在等資料，下一個週期硬體會自動切去跑執行緒 B，所以管線永遠有事做。
  • 缺點：嚴重犧牲了「單一執行緒的效能（Single-thread performance）」。即使全晶片此時只有執行緒 A 在跑，它也必須被迫每隔幾個週期才能發射一次，速度被大幅拖慢。

2. 粗粒度多執行緒 (Coarse-Grained Multithreading)

  • 運作機制：處理器平時死心塌地只跑同一個執行緒。只有當前執行緒遇到重大、昂貴的停頓（例如最致命的 L2/L3 Cache Miss）時，硬體才會觸發切換機制，換去跑另一個執行緒。
  • 優點：單一執行緒在平時能獨佔所有硬體資源，跑出最高速度；且硬體不需要每週期切換。
  • 缺點：存在切換成本（Switching Overhead）。當遇到 Cache Miss 決定切換時，管線必須被清空（Pipeline Flush）或需要幾個週期的啟動延遲（Startup penalty），因此無法隱藏短暫的管線卡頓。

3. 同步多執行緒 (Simultaneous Multithreading, SMT)

  • 運作機制：這是不折不扣的終極進化型。它架構在多重發射超純量（Multiple-Issue Superscalar）處理器之上。它允許在同一個時鐘週期之內（Within the same cycle），同時挑選並發射來自多個不同執行緒的指令進入各個運算單元！
  • 解決的痛點 (Slide 204)：超純量處理器平時會產生兩種閒置空格：
    1. 垂直空缺 (Vertical Waste)：一整個週期都沒有指令可以發射。
    1. 水平空缺 (Horizontal Waste)：某個週期發射了 2 道指令，但剩下的 2 個發射槽是空的。
    • SMT 的神級解法：完美將水平與垂直空缺通通填滿，徹底釋放多核超純量的實體運算威力。

📑計算機結構進階特訓：VLIW 架構與硬體多執行緒（TLP）全圖解教學
💡 Notion 考前核心心法導讀：

現代 CPU 面臨兩大平行度開發範式：

  1. ILP (指令層級平行度)：單一程式內找平行。VLIW 把這個責任完全推給編譯器（軟體）。
  1. TLP (執行緒層級平行度)：多個獨立程式間找平行。Fine-Grained、Coarse-Grained、SMT 則是利用硬體切換不同的執行緒，用別人的指令來填滿自己閒置的運算格子。
🛑 1. VLIW (超長指令字) 架構深度解構

VLIW (Very Long Instruction Word) 的核心哲學是：「硬體退後，編譯器往前站！」 它完全放棄了硬體動態檢查 Hazard 的電路，把所有的排程與相依性檢查全部交給軟體。

⚙️ 運作機制與核心特徵

  • 指令打包 (Issue Packet)：編譯器在編譯程式碼時，會同時審視好幾道指令，找出完全沒有相依性、可以同時執行的操作，把它們塞進一條長度高達 112 到 168 bits 的「超級長指令包」中。
  • 典型封包範例：一個 VLIW 封包內可能同時包含：[Memory Ref 1] [Memory Ref 2] [FP Op 1] [FP Op 2] [Integer/Branch]。
  • 硬體盲目執行：硬體不配備任何動態 Hazard 檢查電路。當硬體吞入這個封包時，它會無條件相信編譯器，直接把裡面的子指令分流送進對應的實體 ALU 執行。

🟢 VLIW 的絕對優勢

  • 硬體成本極低、極省電：消滅了超純量（Superscalar）中極度耗電且複雜的動態發射（Dynamic Issue）、保留站（RS）、CDB 與相依性比較電路。
  • 理想狀態下吞吐量高：如果編譯器非常強大、程式規律（如數位訊號處理 DSP、影像處理），格子能被天天填滿，效能與功耗比會非常驚人。

🔴 VLIW 的致命缺陷（期末考必考簡答）

  • 程式碼體積爆炸（Wasted Bits）：如果程式碼邏輯複雜，編譯器找不到足夠多平行的指令，格子填不滿，就必須在長指令包裡塞滿無用的 NOP (空指令)，浪費極大的儲存空間。
  • 同步鎖定限制 (Lock-Step)：封包內的指令是生死與共的。只要其中一個單元（例如 Memory 管道發生 Cache Miss），全晶片所有其他功能單元通通要被強迫一起停頓（Stall）。
  • 二進位代碼不相容 (Binary Incompatibility)：程式碼是跟著硬體架構客製化的。如果下一代晶片多加了一個加法器，舊的 VLIW 程式就完全不能跑，必須全面重新編譯。

🛑 3. 細粒度多執行緒 (Fine-Grained Multithreading)

⚙️ 運作機制

  • 每週期輪切 (Cycle-by-Cycle)：處理器在每一個時鐘週期（Every Cycle），都會利用輪詢（Round-Robin）等邏輯，強迫管線切換到下一個不同的執行緒（Thread）來抓取並發射指令。

🟢 優缺點分析

  • 👍 優點：完美隱藏短暫卡頓：如果 Thread A 發生了 1~2 週期的資料相依（RAW）或分支延遲，硬體在下個週期反正會去跑 Thread B、C、D，所以這些短暫的 Stalls 會在管線中被完美無痛隱藏。
  • 👎 缺點：毀滅單一執行緒效能 (Single-Thread Performance)：因為硬體每週期都在強迫換人，即使此時全晶片只有 Thread A 渴望運算、其他 Thread 都在閒置，Thread A 也必須被迫在原地乾等，導致單一程式的執行時間被拉得極長。
🛑 4. 粗粒度多執行緒 (Coarse-Grained Multithreading)

⚙️ 運作機制

  • 大難臨頭才換人：處理器平时非常專一，全速只跑同一個執行緒（例如 Thread A）。只有當 Thread A 遭遇到極其昂貴、嚴重的卡頓（例如最致命的 L2/L3 Cache Miss）時，硬體才會觸發切換，換去跑 Thread B。

🟢 優缺點分析

  • 👍 優點：維護單一執行緒效能：在平時沒有發生 Cache Miss 的絕大多數時間裡，當前執行緒可以獨佔全晶片所有的實體硬體資源，跑出全速。
  • 👎 缺點：存在切換成本 (Switching Overhead)：因為平時不切換，一旦大難臨頭要切換時，管線內可能還殘留著舊執行緒的指令，硬體通常需要清空管線（Pipeline Flush）或付出好幾個週期的排空與啟動懲罰（Startup penalty），因此完全無法隱藏短暫的 1~2 週期小卡頓。
🛑 5. SMT (同步多執行緒) 架構——現代 CPU 的終極型態

SMT (Simultaneous Multithreading) 是將「多重發射超純量 (Multiple-Issue Superscalar)」與「多執行緒技術」進行融合的終極架構（Intel 的 Hyper-Threading 便是此技術）。

⚙️ 運作機制

  • 打破週期障礙：它徹底打破了前兩者「單一週期內只能容納同一個執行緒指令」的物理限制。它允許在同一個時鐘週期之內（Within the same clock cycle），同時挑選並混搭發射來自 Thread A、Thread B、Thread C 的指令，並發射進入各個實體運算單元。
  • 完美消除空缺：同時利用 ILP（指令間平行）與 TLP（執行緒間平行），把垂直空缺與水平空缺通通榨乾。

⚙️ 💥 申論題必背：SMT 的硬體資源分配

為了讓 OS 覺得這是一顆獨立的雙核/多核 CPU，硬體內部做了精密的分流：

  • 獨立複製 (Duplicated) 的元件（佔 5% 面積）：PC (程式計數器)、Architectural Registers (架構暫存器檔案狀態)、Rename Table (重命名表)。
  • 完全共享 (Shared) 的電路：實體運算單元 (ALU/加法器)、各級 Caches (L1/L2/L3)、ROB 與保留站水池。

🟢 優缺點分析

  • 👍 優點：投資報酬率極高：晶片只需多花約 5% 的面積去多做幾組 PC 和暫存器，就能為全晶片換來高達 20% ~ 30% 的整體吞吐量（Throughput）提升！
  • 👎 缺點：資源激烈爭奪（Contention）：因為快取和 ALU 是共享的，如果 Thread A 和 Thread B 同時在瘋狂存取記憶體，會導致快取污染，互相拖累；此外也容易引發快取側信道（Cache Side-Channel）的安全性風險（如 Spectre 漏洞）。
